# PostgreSQL Extension on Pure C# NativeAOT

Proof-of-concept: PostgreSQL C-language extension written entirely in C# using .NET NativeAOT compilation. No C code, no wrappers — all PostgreSQL ABI structures replicated in C# via `[StructLayout]`.

## Results

| Build | DLL Size | Toolchain |
|-------|----------|-----------|
| .NET 10 NativeAOT (baseline) | 723 KB | `dotnet publish` + MSVC linker |
| NativeAOT + all ILC optimizations | 694 KB | same + `IlcDisableReflection`, `IlcFoldIdenticalMethodBodies`, etc. |
| **bflat zerolib** | **6.5 KB** | `bflat build --stdlib:zero -Os` |

All three builds produce the same result and pass all tests.

```
postgres=# SELECT csharp_add(2, 3);
 csharp_add
------------
          5

postgres=# SELECT csharp_add(2147483600, 47);
 csharp_add
------------
 2147483647
```

## How It Works

PostgreSQL loads C extensions via `LoadLibrary` (Windows) / `dlopen` (Linux) and expects three types of exported symbols:

1. **`Pg_magic_func`** — returns a pointer to `Pg_magic_struct` for ABI compatibility verification. PG checks `len`, `version`, `funcmaxargs`, etc. against its own values via `memcmp`.

2. **`pg_finfo_<funcname>`** — returns `Pg_finfo_record { api_version = 1 }`, declaring version-1 calling convention.

3. **`<funcname>`** — the actual function, taking `FunctionCallInfo` (pointer to `FunctionCallInfoBaseData`) and returning `Datum` (`nint` / `uintptr_t`).

In C, these are generated by macros (`PG_MODULE_MAGIC`, `PG_FUNCTION_INFO_V1`). In this project, we define the same binary-compatible structs in C# using `[StructLayout(LayoutKind.Explicit)]` with exact field offsets, and export functions via `[UnmanagedCallersOnly(EntryPoint = "...")]`.

### Key Struct Layouts (PG 18, x64)

**Pg_magic_struct** (72 bytes):
```
offset  0: int len             = 72           (sizeof)
offset  4: int version         = 1800         (PG_VERSION_NUM / 100)
offset  8: int funcmaxargs     = 100          (FUNC_MAX_ARGS)
offset 12: int indexmaxkeys    = 32           (INDEX_MAX_KEYS)
offset 16: int namedatalen     = 64           (NAMEDATALEN)
offset 20: int float8byval     = 1            (FLOAT8PASSBYVAL)
offset 24: byte[32] abi_extra  = "PostgreSQL"  (FMGR_ABI_EXTRA)
offset 56: nint name           = NULL
offset 64: nint version_str    = NULL
```

**FunctionCallInfoBaseData** (32 bytes fixed + flexible `args[]`):
```
offset  0: nint flinfo        (8)   FmgrInfo pointer
offset  8: nint context       (8)   call context
offset 16: nint resultinfo    (8)   result info
offset 24: uint fncollation   (4)   collation OID
offset 28: byte isnull        (1)   result null flag
offset 30: short nargs        (2)   argument count
offset 32: NullableDatum[]          args array (each 16 bytes)
```

**NullableDatum** (16 bytes):
```
offset 0: nint value   (8)   Datum = uintptr_t
offset 8: byte isnull  (1)   null flag + 7 bytes padding
```

## Build & Install

### Option A: bflat zerolib (6.5 KB)

Prerequisites: [bflat](https://github.com/bflattened/bflat) (`winget install bflat`), PostgreSQL 18.

```powershell
# Build
bflat build PgExtension.Bflat/PgExtension.cs --stdlib:zero --target Shared `
  -o .build/pg_dotnet.dll -Os --no-reflection --no-stacktrace-data `
  --no-globalization --no-exception-messages --no-debug-info

# Install (requires admin)
copy .build\pg_dotnet.dll "C:\Program Files\PostgreSQL\18\lib\"
```

### Option B: .NET 10 NativeAOT (694 KB)

Prerequisites: .NET 10 SDK, Visual Studio (for MSVC linker), PostgreSQL 18.

```powershell
# Build (add vswhere.exe to PATH if needed)
$env:PATH += ";C:\Program Files (x86)\Microsoft Visual Studio\Installer"
dotnet publish PgExtension.Aot -r win-x64 -c Release -p:NativeLib=Shared

# Install (requires admin)
copy PgExtension.Aot\bin\Release\net10.0\win-x64\publish\pg_dotnet.dll "C:\Program Files\PostgreSQL\18\lib\"
```

### Register & Test (same for both)

```powershell
psql -U postgres -p 5433 -d postgres -c "
  CREATE OR REPLACE FUNCTION csharp_add(integer, integer) RETURNS integer
    AS 'pg_dotnet', 'pg_add' LANGUAGE C STRICT;
"
psql -U postgres -p 5433 -d postgres -c "SELECT csharp_add(2, 3);"
```

## Version Compatibility

The struct layouts and constants are tied to **PostgreSQL 18**. Key differences from PG 17:

- PG 17: flat `Pg_magic_struct` (56 bytes) — `len` + 5 ints + `char[32]`
- PG 18: nested struct (72 bytes) — `len` + `Pg_abi_values` + `name` pointer + `version` pointer

`FunctionCallInfoBaseData` and `NullableDatum` are identical across PG 17–18.

When upgrading PostgreSQL major version, check `fmgr.h` for struct changes and update constants in `PgExtension.cs` (primarily `PgVersion`).

## Roadmap

### 1. DLL Size Optimization — DONE

Achieved **100x reduction**: 723 KB → 6.5 KB using [bflat](https://github.com/bflattened/bflat) with `--stdlib:zero`.

| Approach | Size | Why |
|----------|------|-----|
| .NET 10 NativeAOT baseline | 723 KB | Full runtime: GC, exceptions, threading, type metadata |
| + all ILC flags (`IlcDisableReflection`, `IlcFoldIdenticalMethodBodies`, etc.) | 694 KB | Runtime code is incompressible floor |
| **bflat `--stdlib:zero`** | **6.5 KB** | No GC, no runtime — just native code |

Key insight: standard NativeAOT's ~700 KB floor is the .NET runtime (GC + exception handling + threading). For functions that don't allocate heap memory, bflat's zerolib eliminates all of this. Trade-off: no GC means no `string`, no `List<T>`, no heap allocations — only stack/static data and raw pointers.

References:
- https://github.com/bflattened/bflat
- https://medium.com/@MStrehovsky/building-a-self-contained-game-in-c-under-8-kilobytes-74c3cf60ea04

### 2. Benchmarking

Compare C# extension performance against an equivalent pure C extension:

- Call overhead: latency of a single `SELECT csharp_add(2, 3)` vs C equivalent.
- Throughput: sustained calls/sec under `pgbench` load.
- First-call cost: DLL load + initialization time.
- NativeAOT vs bflat: compare both builds against each other and against C.

### 3. Complex Data Types — Tables and Composite Structures

Working with PostgreSQL tables and complex types from C#:

- **text/varchar**: `PG_GETARG_TEXT_PP` → `struct varlena` with 4-byte length header + data. Need to define `varlena` layout in C# and handle TOAST decompression (or call PG's `pg_detoast_datum_packed` via `[DllImport]` from `postgres.exe`).
- **SRF (Set-Returning Functions)**: return multiple rows using `SRF_IS_FIRSTCALL`, `SRF_PERCALL_SETUP`, `SRF_RETURN_NEXT` protocol. Requires accessing `FuncCallContext` from `resultinfo`.
- **Composite types / tuples**: build `HeapTuple` via PG's `heap_form_tuple`. Would require importing PG internal functions.
- **Arrays**: PostgreSQL array type uses `ArrayType` header. Can be deconstructed via `deconstruct_array`.
- **Approach**: for complex types, a hybrid model may work best — export the business logic from C# as simple primitives/pointers, and use a thin C shim only for PG type marshaling. Or import PG helper functions into C# via `[DllImport("postgres")]`.

### 4. Network and External Calls from Extension

Making HTTP/network calls from within a PG extension:

- **HttpClient**: `System.Net.Http.HttpClient` works with NativeAOT but adds significant binary size (~2-5 MB for TLS stack). Need `System.Net.Security` and platform TLS (SChannel on Windows).
- **Use case**: call external APIs (geocoding, ML inference, data enrichment) directly from SQL queries.
- **Risks**:
  - Network calls block the PG backend process. Long timeouts can stall queries and consume connection slots.
  - PG signal handling (cancel, terminate) may conflict with .NET async/socket operations.
  - Memory: .NET GC allocations during HTTP calls may pressure the PG process.
- **Mitigation**: use synchronous calls with strict timeouts. Consider `background worker` process instead of in-query calls for heavy operations.
- **Example target**: `SELECT http_get('https://api.example.com/data')` returning JSON as `text`.
